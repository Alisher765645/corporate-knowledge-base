# PRD: Корпоративная база знаний по нормативным документам

**Статус:** черновик для дальнейшей проработки
**Версия:** 0.1

---

## 1. Обзор продукта

Внутреннее корпоративное приложение для поиска по нормативным документам компании. Пользователь задаёт вопрос или ключевые слова текстом, голосом или фото — система находит релевантные фрагменты документов и формирует ответ со ссылками на первоисточник. Документы в систему загружают только уполномоченные администраторы; рядовые сотрудники не могут пополнять базу.

## 2. Проблема

Сотрудники тратят время на поиск нужных норм в разрозненных папках и документах, не всегда знают точные формулировки для поиска и часто формулируют запрос вопросом, а не набором ключевых слов. Нужен единый, быстрый и терпимый к ошибкам инструмент поиска на двух языках — русском и казахском.

## 3. Цели

- Сократить время поиска нужной нормы до секунд.
- Дать возможность искать как по ключевым словам, так и естественным вопросом.
- Полноценно поддерживать русский и казахский язык — интерфейс, ввод, ответ.
- Гарантировать, что ответы опираются только на реальные документы компании (без "выдумывания" норм).
- Сохранить лаконичный, ненагруженный интерфейс.

## 4. Целевая аудитория

| Роль | Описание |
|---|---|
| Сотрудник | Основной пользователь, только читает/ищет, не может загружать документы |
| Администратор контента | Загружает, обновляет, архивирует документы (HR, юридический департамент, комплаенс) |

## 5. Границы MVP

**В рамках:**
- Поиск по ключевым словам и по вопросу
- Текстовый, голосовой и фото-ввод запроса
- Ограничение ввода — 5000 символов
- Обработка запросов с опечатками и ошибками
- Ответы на русском и казахском с указанием источника

**Вне рамок (на будущее):**
- Самостоятельная загрузка документов сотрудниками
- Мобильное приложение (если не указано иное — сначала веб)
- Кросс-язычный поиск "запрос на одном языке → документ на другом" (уточнить приоритет)
- Workflow согласования документов перед публикацией

## 6. Функциональные требования

### 6.1 Управление контентом (администратор)
- **FR1.** Загружать, обновлять и архивировать документы может только роль "администратор контента".
- **FR2.** Поддерживаемые форматы исходных документов — уточнить (PDF, DOCX, XLSX как минимум).
- **FR3.** Версионирование документов и история изменений.

### 6.2 Поиск и вопросы
- **FR4.** Поиск по ключевым словам.
- **FR5.** Поиск по вопросу на естественном языке с ответом, сгенерированным по найденным фрагментам (RAG).
- **FR6.** Гибридный поиск — комбинация векторного (семантического) и полнотекстового поиска.
- **FR7.** Каждый ответ содержит ссылку на исходный документ и, по возможности, конкретный раздел/пункт.
- **FR8.** Система не формирует ответ, не подкреплённый найденными документами — при отсутствии релевантных фрагментов явно сообщает об этом, а не додумывает.

### 6.3 Способы ввода запроса
- **FR9.** Текстовый ввод с лимитом 5000 символов и счётчиком символов в интерфейсе.
- **FR10.** Голосовой ввод (speech-to-text) на русском и казахском языках.
- **FR11.** Ввод через фото — распознавание текста с изображения (OCR) для последующего поиска; фото используется только как способ задать запрос, не для пополнения базы документов.
- **FR12.** Лимит в 5000 символов применяется одинаково к тексту, к расшифровке голоса и к результату OCR.

### 6.4 Обработка ошибок и качество запроса
- **FR13.** Автоматическое исправление опечаток до выполнения поиска.
- **FR14.** Автоматическое определение языка запроса.
- **FR15.** Нормализация/переформулирование запроса перед поиском для повышения точности.
- **FR16.** Ответ формируется на языке запроса.

### 6.5 Мультиязычность
- **FR17.** Интерфейс полностью локализован на русский и казахский.
- **FR18.** Документы могут быть на любом из двух языков; поиск должен одинаково хорошо находить их независимо от языка запроса.

## 7. Нефункциональные требования

- **NFR1.** Лаконичный минималистичный UI: одно центральное поле ввода как основной элемент, минимум настроек на первом экране.
- **NFR2.** Целевое время ответа — уточнить (ориентир: до 5 секунд для большинства запросов).
- **NFR3.** Ролевая модель доступа; при необходимости — разграничение документов по конфиденциальности/департаментам.
- **NFR4.** Размещение данных — открытый вопрос, см. раздел 10.
- **NFR5.** Аудит-лог поисковых запросов — рекомендуется, так как документы носят нормативный характер.
- **NFR6.** Платформы доступа — уточнить (веб обязателен, мобильная версия — по необходимости).

## 8. Пользовательские сценарии

| Сценарий | Шаги |
|---|---|
| UF1. Текстовый вопрос | Сотрудник вводит вопрос → нормализация → гибридный поиск → ответ со ссылками |
| UF2. Голосовой запрос | Сотрудник надиктовывает вопрос → распознавание речи → далее как UF1 |
| UF3. Запрос по фото | Сотрудник фотографирует фрагмент документа → OCR → далее как UF1 |
| UF4. Загрузка документа | Администратор загружает документ → обработка (чанкинг, эмбеддинги) → документ доступен для поиска |

## 9. Архитектура (высокий уровень)

1. **Загрузка документов** — доступна только администратору.
2. **Обработка и индексация** — чанкинг текста, генерация эмбеддингов, запись в базу знаний (векторный + полнотекстовый индекс).
3. **Ввод запроса** — текст / голос (speech-to-text) / фото (OCR), с общим лимитом 5000 символов.
4. **Нормализация запроса** — исправление опечаток, определение языка, переформулирование.
5. **Гибридный поиск** — по базе знаний (вектор + ключевые слова).
6. **LLM + RAG** — формирование ответа строго по найденным фрагментам, со ссылками на источники.
7. **Пользователь** — получает ответ.

## 10. Открытые вопросы

- **Размещение системы:** облако (внешние API) или приватный/локальный контур — определяет выбор моделей эмбеддингов, LLM, речевого движка и OCR. Критично решить в первую очередь, так как документы являются внутренними нормативными.
- Какие форматы исходных документов должны поддерживаться в первую очередь?
- Нужен ли кросс-язычный поиск (вопрос на казахском находит документ на русском и наоборот)?
- Нужен ли workflow согласования документа перед публикацией в базе?
- Нужна ли мобильная версия в MVP или достаточно веба?
- Целевые SLA по времени ответа и доступности.

## 11. Метрики успеха

- Доля запросов, получивших релевантный ответ (по обратной связи 👍/👎 от пользователей).
- Среднее время получения ответа.
- Доля сотрудников, использующих систему еженедельно.
- Снижение числа типовых обращений в HR/юридический отдел по вопросам, покрытым базой.

## 12. Черновая дорожная карта

| Этап | Содержание |
|---|---|
| MVP 1 | Текстовый поиск и Q&A на русском, загрузка документов администратором, гибридный поиск |
| MVP 2 | Полноценная поддержка казахского языка (интерфейс, поиск, ответы) |
| MVP 3 | Голосовой и фото-ввод запроса |
| MVP 4 | Ролевой доступ, аудит-лог, аналитика использования |
